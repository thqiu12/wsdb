### 2A.1 Receipt Numbering

Receipt numbers should be generated using:

- Issue date
- Admission period
- Sequence number
- Optional campus/school code

Example pattern:

```text
RC-20260420-202610-0001
```

Final receipt format still needs school confirmation.

Other confirmed numbering rules:

```text
Student number: YYYYMM + random 3 digits
Certificate number: YYYYMMDD + random 3 digits
```

### 2A.2 Agent-Collected Payment

When an agent collects payment:

- Receipt recipient/legal target remains tied to the student/applicant record.
- Display name on receipt/payment record should show the agent name.
- Student name should still be referenced in the receipt details.
- Agent collection must be searchable and auditable.

### 2A.3 Receipt Void/Reissue

Receipt operations:

- Issue
- Void
- Reissue

Void requires:

- Reason
- User
- Timestamp

Reissue must not overwrite the old receipt. It creates a new version or new receipt number according to school rule.

### 2A.4 Full COE Release

Full COE must not be released until:

- Full tuition payment is confirmed.
- Required receipt has been issued.

Any override must require high-level permission and audit log.

### 2.2 Interview Status

Interview statuses should include:

```text
passed
hold
re_interview
failed
```

For display:

- `合格`
- `保留`
- `复试`
- `不合格`

The system should support multiple interview records for the same applicant.

### 2.3 Acceptance Notice Templates

Acceptance notice generation must select template by:

- School
- Campus
- Admission term, if needed
- Language, if needed
- Template version

Template examples:

- Takadanobaba campus acceptance notice
- Other campus acceptance notices

### 2.4 COE Submission Deadline

The system must support both:

1. Term-level deadline
   - Example: all 2026 October intake students share one target deadline.

2. Student-level individual deadline
   - Needed for school corporation / rolling submission workflows.
   - Individual deadline overrides term-level deadline.

### 2.5 COE Release Control

Before tuition payment:

- Staff can upload full COE.
- Staff can create/send partial screenshot to student.
- Full COE remains restricted.

After required tuition payment:

- Staff can issue receipt.
- Staff can send full COE.

Open confirmation:

- Who can override full COE release restriction, if override is allowed at all.

## 4. Permission Rules

| Operation | Role / Permission Holder | Status |
|---|---|---|
| Modify student basic information | staff | Confirmed |
| View/download passport, residence card, COE | staff | Confirmed |
| Run AI check | staff | Confirmed |
| Confirm AI check | staff | Confirmed |
| Generate immigration files | immigration report staff | Confirmed |
| Mark immigration report as submitted | immigration report staff | Confirmed |
| Confirm tuition/payment receipt | staff | Confirmed |
| Issue receipt | staff | Confirmed |
| Send full COE | staff | Confirmed |
| Void certificate/receipt | staff | Confirmed |
| Create new user account | manager | Confirmed |

### 4.1 Role Definitions

`staff`

- Daily office staff role.
- Can manage student/applicant information, files, payments, receipts, AI checks, COE release, and certificate/receipt voiding.

`immigration_report_staff`

- Staff member responsible for immigration report operations.
- Can generate immigration files and mark reports as submitted.

`manager`

- Management/admin role.
- Can create new user accounts and manage user access.

### 4.2 Permission Notes

- Even when `staff` can view/download passport, residence card, and COE, all downloads must be logged.
- Sending full COE requires full tuition confirmation according to confirmed payment rules.
- Voiding certificate/receipt must require reason, user, and timestamp.
- Immigration file generation and submission marking should be restricted to the immigration report staff permission.
